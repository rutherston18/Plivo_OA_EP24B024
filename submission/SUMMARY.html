2. The problem & the constraints

Train a byte-level-scorable language model on ~7 MB of mixed English+Hindi under hard caps: ≤2000 optimizer steps≤2,000,000 paramsCPU only pure PyTorch + numpy + stdlibonly train_corpus.txt no tiktoken / rustbpe / flash-attn / custom kernels. The scorer runs python evaluate.py --checkpoint ckpt.pt --text_file <file> and the tokenizer must be lossless.

Because steps and params are capped, the game is sample efficiency (get the most out of 2000 steps) and parameter efficiency (spend the 2M budget where it matters), plus a tokenizer that compresses Devanagari (3–4 bytes/char) so a fixed context covers more text and each predicted token covers more bytes.

4. Every improvement — what it does & why it helps

Trapezoidal LR schedule (warmup→const→linear warmdown) training E1

Replaces the baseline's constant LR. With only 2000 steps, warmup lets you use a high peak LR safely and the warmdown phase is where loss drops fastest.

AdamW + decoupled weight decay + gradient clipping training E1

Decoupled WD regularizes the 2-D weights; clipping stabilizes the high-LR training the step budget demands.

RoPE (rotary positions) architecture E2

Relative positions injected in attention; removes the learned position table (frees params) and generalizes better across positions.

Parameter-free RMSNorm architecture E2

Cheaper and more stable than LayerNorm; no learnable affine params to spend budget on.

QK-norm architecture E2

Normalizes queries/keys before attention, preventing logit blow-ups at high LR.

ReLU² MLP + zero-init output projections + logit softcap architecture E2

ReLU² is a strong cheap activation; zero-init projections make each residual block an identity at init (clean start ≈ ln(V) loss); softcap keeps early logits bounded.

Muon optimizer (Newton–Schulz orthogonalized updates) optimizer E3

Orthogonalizes each 2-D gradient update, which converges faster per step than Adam — exactly what a fixed 2000-step budget rewards. Embeddings/head stay on AdamW.

Byte-level BPE-1024 tokenizer (pure Python, lossless) + weight tying tokenizer E4 (done)

Compresses multi-byte Devanagari into single tokens (dev: 2.33 bytes/token, Hindi 4.0). Since BPB is per-byte, each token covering more bytes directly lowers the score, and a fixed 128-token context now spans ~300 bytes. Tying input/output embeddings frees ~164k params to reinvest.

SwiGLU MLP architecture E5 (planned)

SwiGLU gives better capacity per parameter than a plain 4× MLP.

Hash n-gram / Engram embeddings family 2 E6 (planned)

Cheap hashed bigram/trigram lookups injected into the stream give tiny models instant local spelling/morphology memory, freeing attention/depth for higher-level structure.

Selective State-Space Model backbone (Mamba-1) family 3 E7 (priority)

Linear-time O(N) recurrence instead of O(N²) attention — cache-friendly on memory-bandwidth-bound CPUs; lets us boost hidden dim and max out the 2M param budget.

Hierarchical / dynamic patching (BLT / SpaceByte) family 1 E8 (stretch)

Group bytes into patches so heavy layers see shorter sequences. Deprioritized: BPE already captures most of this benefit, losslessly.

5. Weights & Biases logging & reproducibility

train.py integrates wandb (--wandb, project plivo-llm-speedrun). Per step it logs train/loss, lr_mult and grad_norm; with --eval_every it logs val/bpb vs step (on a cheap dev subset, with a full dev eval at the end) so the BPB-vs-step curve is visible live. This makes the effect of each aggressive-LR variant observable without waiting for the run to finish.

Reproducibility: every run started with --tag <name> snapshots the exact model.py, train.py, muon.py, tokenizer.py, evaluate.py, bpe.json + ckpt.pt + run_meta.json (args & config) into experiments/<name>/, so any checkpoint can be rebuilt or resumed.

This project was built with an AI coding assistant (Cursor) under continuous human direction. The human set the strategy and priorities (which repos/ideas to port, "build an SSM and max out params", "train more aggressively", the ask-before-full-runs and archive-everything workflow, and all go/no-go decisions). The assistant did the implementation: reading the repos, porting Muon/RoPE/SSM/BPE into pure PyTorch under the caps, wiring wandb + archiving, running the experiments, and writing this report. Every architectural change was validated by the official scorer, and each run's hypothesis→result→conclusion is logged in RUNLOG.md (reproduced in the appendix).

Appendix — experiments in detail

Baseline — dev BPB 2.3718 (1,339,840 params)

Unmodified starter code. The reference every experiment is measured against.

E1 — dev BPB 2.1864 -7.8% vs baseline

Hypothesis. The baseline is under-trained: constant LR, no warmup/decay, plain Adam.

Changed. AdamW (wd 0.1 on 2-D weights), trapezoidal LR (100-step warmup, warmdown over last 40% to 0), peak LR 3e-4→1e-3, grad clip 1.0. Architecture unchanged.

Result. dev BPB 2.1864 (1,339,840 params, 2000 steps).

Conclusion. Big cheap win from optimization alone; loss falls fastest during warmdown.

E2 — dev BPB 2.0421 -13.9% vs baseline

Hypothesis. The starter block (learned pos, LayerNorm, GELU, biases) is dated.

Changed. RoPE (drop learned pos), parameter-free RMSNorm, QK-norm, ReLU², bias=False, zero-init output projections, logit softcap 15.

Result. dev BPB 2.0421 (1,310,720 params, 2000 steps).

Conclusion. Modernization helps and is 'free' on params (fewer than baseline).

E3 — dev BPB 1.9516 -17.7% vs baseline

Hypothesis. At a fixed 2000-step budget, per-step efficiency matters most.

Changed. Pure-PyTorch single-device Muon (NS-5, momentum 0.95, lr 0.02) for block matrices; AdamW (lr 1e-3) for embeddings+head.

Result. dev BPB 1.9516 (1,310,720 params, 2000 steps).

Conclusion. Muon converges faster per step. Under 2.0 BPB for the first time.

E4 — dev BPB 1.8583 -21.7% vs baseline

Hypothesis. Byte tokenizer wastes 3-4 tokens per Devanagari char; BPB is per-byte so compression directly lowers the score.

Changed. Pure-Python lossless BPE-1024 (dev 2.33 bytes/token) + tied input/output embeddings (frees ~164k params). Muon.

Result. dev BPB 1.8583 (1,392,640 params, 2000 steps).

Conclusion. Largest single win so far; a 128-token context now spans ~300 bytes.
